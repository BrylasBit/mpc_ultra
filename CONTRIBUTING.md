# Contributing to MPC ULTRA v8

Thank you for your interest in contributing! This document provides guidelines for contributing to the project.

## 📋 Code of Conduct

- Be respectful and inclusive
- Provide constructive feedback
- Help others learn and grow
- Report issues responsibly

## 🐛 Reporting Bugs

Found a bug? Please create an issue with:
- **Description**: What's the problem?
- **Steps to Reproduce**: How can we recreate it?
- **Expected vs Actual**: What should happen vs what does?
- **Environment**: Browser, OS, audio system
- **Screenshots/Logs**: If applicable

## 💡 Suggesting Features

Have an idea? Share it!
- **Title**: Clear, concise feature name
- **Description**: What problem does it solve?
- **Use Case**: Who benefits and how?
- **Example**: Mockup or description
- **Alternatives**: Any other approaches?

## 🔧 Development & Pull Requests

### Setting Up
```bash
git clone https://github.com/BrylasBit/mpc_ultra.git
cd mpc_ultra
```

### Making Changes
1. Create a feature branch: `git checkout -b feature/amazing-feature`
2. Make your changes in `index.html`
3. Test thoroughly in multiple browsers
4. Commit with clear messages: `git commit -m "Add: amazing feature"`
5. Push to your fork: `git push origin feature/amazing-feature`
6. Create a Pull Request with description

### Code Guidelines
- Keep code well-formatted and readable
- Use meaningful variable names
- Add comments for complex logic
- Test on Chrome, Firefox, Safari, Edge
- Ensure no console errors

### Types of Contributions
✅ Bug fixes
✅ Performance improvements
✅ New features
✅ Documentation improvements
✅ UI/UX enhancements
✅ Accessibility improvements

## 📝 Commit Message Format

```
Type: Brief description

Detailed explanation if needed:
- What changed
- Why it changed
- Any gotchas
```

**Types:**
- `fix:` - Bug fixes
- `feat:` - New features
- `refactor:` - Code refactoring
- `docs:` - Documentation
- `style:` - Formatting/style changes
- `perf:` - Performance improvements

## 🎨 Code Style

### JavaScript
- Use modern ES6+ features
- Clear, descriptive variable names
- Comments for non-obvious logic
- Avoid one-liners (readability over brevity)

### CSS
- Use CSS variables for consistency
- Follow existing naming conventions
- Comments for complex selectors
- Mobile-first approach

## ✅ Testing Checklist

Before submitting a PR:
- [ ] Tested in Chrome/Edge
- [ ] Tested in Firefox
- [ ] Tested in Safari
- [ ] No console errors
- [ ] No broken features
- [ ] MIDI controls work (if modified)
- [ ] Keyboard controls work (if modified)
- [ ] Sample loading works (if modified)

## 🔐 Security Issues

Found a security vulnerability? **Please don't** open a public issue.
Contact the maintainer privately at: brylas.bit [at] github.com
